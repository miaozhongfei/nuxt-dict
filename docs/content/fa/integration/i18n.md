# بین‌المللی‌سازی i18n

**هدف**: استفاده از nuxt-dict همراه با @nuxtjs/i18n به طوری که داده‌های دیکشنری به طور خودکار هنگام تغییر زبان کاربران همگام‌سازی شوند.

## چه زمانی به این ویژگی نیاز دارید؟

- پروژه شما قبلاً از @nuxtjs/i18n برای مسیریابی چندزبانه و ترجمه‌های UI استفاده می‌کند
- می‌خواهید گزینه‌های dropdown (جنسیت، وضعیت و غیره) به طور خودکار با تغییر زبان کاربر به زبان صحیح تغییر کنند
- داده‌های دیکشنری باید بدون بازنشانی صفحه به‌روزرسانی شوند

## همگام‌سازی غیرفعال (آماده استفاده)

نام cookie پیش‌فرض nuxt-dict برابر با `i18n_redirected` است — دقیقاً همان cookie پیش‌فرض @nuxtjs/i18n. تا زمانی که `locale.source: 'cookie'` (پیش‌فرض) را حفظ کنید، هر دو ماژول به طور خودکار اطلاعات زبان یکسانی را به اشتراک می‌گذارند.

**این به چه معناست؟**

زمانی که کاربر برای اولین بار `/en/about` را بازدید می‌کند:
۱. @nuxtjs/i18n زبان `en-US` را از URL تجزیه کرده و در cookie `i18n_redirected` می‌نویسد
۲. nuxt-dict زبان را از همان cookie می‌خواند و داده‌های دیکشنری را با `lang=en-US` درخواست می‌کند
۳. API دیکشنری داده‌های انگلیسی را برمی‌گرداند

زمانی که کاربر صفحه را بازنشانی می‌کند یا به صفحه جدید می‌رود:
۱. cookie قبلاً شامل `en-US` است
۲. nuxt-dict مستقیماً از این زبان استفاده می‌کند — نیازی به پیکربندی اضافی نیست

## همگام‌سازی فعال (تغییر زبان در زمان اجرا بدون بازنشانی صفحه)

**این قابلیت در nuxt-dict تعبیه شده است — نیازی به کد اضافی نیست.**

زمانی که @nuxtjs/i18n زبان را تغییر می‌دهد (مثلاً کاربر روی دکمه تغییر زبان کلیک می‌کند):

```
@nuxtjs/i18n زبان را تغییر می‌دهد
  → cookie i18n_redirected = 'en-US' را می‌نویسد
    → watch cookie در nuxt-dict تغییر را تشخیص می‌دهد
      → DictManager.setLocale('en-US') → کش را پاک می‌کند
        → watch در useDict / useDictTree فعال می‌شود
          → به طور خودکار داده‌های دیکشنری را با زبان جدید دریافت می‌کند
```

**کاربران نیازی به نوشتن هیچ کد `watch` یا `refresh()` ندارند** — تمام کامپوننت‌های `useDict`، `useDictTree` و `useDictOptions` به طور خودکار بارگذاری مجدد می‌شوند.

## مثال پیکربندی

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@lacqjs/nuxt-dict', '@nuxtjs/i18n'],

  // پیکربندی nuxt-dict
  dict: {
    api: { baseURL: '', dictEndpoint: '/api/dict/list' },
    locale: {
      source: 'cookie',             // پیش‌فرض، نیازی به تغییر نیست
      cookieKey: 'i18n_redirected', // پیش‌فرض، cookie یکسان با @nuxtjs/i18n را به اشتراک می‌گذارد
      default: 'zh-CN',
    },
  },

  // پیکربندی @nuxtjs/i18n
  i18n: {
    locales: ['zh-CN', 'en-US', 'ja-JP'],
    defaultLocale: 'zh-CN',
    strategy: 'prefix_except_default',
  },
})
```

## رفتار تحت استراتژی‌های مختلف

| استراتژی @nuxtjs/i18n | URL در تغییر زبان عوض می‌شود | useDict دوباره دریافت می‌کند |
|------------------------|:--:|:--:|
| `prefix_except_default` (پیش‌فرض) | بله (`/about` → `/en/about`) | ✅ خودکار |
| `prefix` | بله (`/zh/about` → `/en/about`) | ✅ خودکار |
| `no_prefix` | خیر | ✅ خودکار (از طریق cookie watch) |

صرف‌نظر از استراتژی، داده‌های دیکشنری همیشه همگام می‌شوند — تحت استراتژی‌های `prefix_*`، تغییر مسیر باعث بازسازی کامپوننت می‌شود (دریافت مجدد طبیعی)؛ تحت `no_prefix`، cookie watch به عنوان پشتیبان عمل می‌کند.

## نکات

> مطمئن شوید که `locale.cookieKey` در nuxt-dict با `detection.cookieKey` در @nuxtjs/i18n **مطابقت دارد**. اگر یکی را تغییر دهید، دیگری را نیز به‌روز کنید.

> دیکشنری‌های درختی (`useDictTree`) نیز در تغییر زبان به طور خودکار دوباره دریافت می‌شوند، اما **گره انتخاب شده فعلی به طور خودکار دوباره انتخاب نمی‌شود**. اگر cascader شما مقادیر را به صورت code ذخیره می‌کند، برچسب‌های ترجمه شده به طور خودکار به‌روز می‌شوند. اگر مقادیر متن label باشند، ممکن است پس از تغییر زبان به مدیریت اضافی نیاز داشته باشید.

> در طول پیش‌واکشی SSR، سرور داده‌ها را با استفاده از زبان درخواست اولیه دریافت می‌کند. اگر زبان کلاینت پس از hydration متفاوت باشد، داده‌ها به طور خودکار دوباره دریافت می‌شوند. نمایش لحظه‌ای زبان اشتباه در اولین rendering بخشی از فرآیند عادی hydration است.

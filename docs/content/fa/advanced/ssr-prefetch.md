**هدف**: درک نحوه کار پیش‌واکشی SSR و پیکربندی نوع‌های دیکشنری که باید پیش‌واکشی شوند برای بهبود عملکرد اولین نمایش.

## چرا پیش‌واکشی نیاز است؟

به طور پیش‌فرض، جریان بارگذاری داده‌های دیکشنری به این صورت است:

```
کاربر صفحه را باز می‌کند ← سرور HTML را رندر می‌کند ← HTML به مرورگر ارسال می‌شود
  ← مرورگر HTML را نمایش می‌دهد ← Vue مسئولیت را می‌گیرد (hydration)
    ← useDict در onMounted درخواست داده ارسال می‌کند ← بارگذاری کامل ← به‌روزرسانی صفحه
```

مشکل: بین hydration و اتمام بارگذاری داده‌ها، کاربر **وضعیت loading** یا **کد اصلی** را می‌بیند.

پس از فعال‌سازی پیش‌واکشی:

```
کاربر صفحه را باز می‌کند ← سرور HTML را رندر می‌کند
  ← همزمان: سرور داده‌های دیکشنری را پیش‌واکشی کرده و در payload HTML تزریق می‌کند
  ← مرورگر HTML را نمایش می‌دهد (از قبل شامل متن ترجمه شده)
    ← Vue مسئولیت را می‌گیرد، مستقیماً داده‌ها را از payload به کش حافظه می‌خواند
```

## پیکربندی پیش‌واکشی

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@lacqjs/nuxt-dict'],
  dict: {
    ssr: {
      prefetch: ['gender', 'status', 'region', 'industry'],
    },
  },
})
```

`prefetch` یک آرایه رشته‌ای است که نوع‌های دیکشنری که باید در سمت سرور پیش‌واکشی شوند را فهرست می‌کند. ماژول در زمان رندر سمت سرور، به طور همزمان داده‌های تمام این دیکشنری‌ها را دریافت می‌کند.

## ترکیب پیش‌واکشی و بارگذاری درخواستی

پیش‌واکشی "بارگذاری کامل" نیست، بلکه "داده‌های ضروری برای اولین نمایش提前 بارگذاری می‌شوند". سایر دیکشنری‌ها همچنان درخواستی بارگذاری می‌شوند:

```vue
<template>
  <div>
    <!-- gender در لیست prefetch است ← سرور بارگذاری کرده، اولین نمایش مستقیم نتیجه ترجمه -->
    <p>جنسیت: {{ $dict.translate('gender', user.gender) }}</p>

    <!-- industry هم در prefetch است ← اولین نمایش مستقیم قابل استفاده -->
    <p>صنعت: {{ $dict.translate('industry', user.industry) }}</p>

    <!-- detail-type در prefetch نیست ← درخواستی بارگذاری می‌شود -->
    <div v-if="showDetail">
      <p>نوع جزئیات: {{ $dict.translate('detail_type', item.type) }}</p>
    </div>
  </div>
</template>
```

## تحمل خطای پیش‌واکشی

پیش‌واکشی از `Promise.all` برای درخواست‌های همزمان استفاده می‌کند، اما **شکست جزئی را مجاز می‌داند**:

```ts
// پیاده‌سازی داخلی ماژول
async function executePrefetch(manager, types) {
  try {
    await Promise.all(types.map(type => manager.getDict(type)))
  } catch {
    // شکست پیش‌واکشی تکی فقط یک هشدار ثبت می‌کند، رندر صفحه را مسدود نمی‌کند
    logger.warn('SSR prefetch failed:', e)
  }
}
```

یعنی حتی اگر پیش‌واکشی یک دیکشنری خاص شکست بخورد، صفحه همچنان به طور عادی رندر می‌شود. فقط داده‌های آن دیکشنری در سمت کلاینت درخواستی بارگذاری می‌شود.

## تأثیر عملکرد

- **تأثیر مثبت**: داده‌های دیکشنری اولین نمایش بخشی از payload HTML می‌شوند، کلاینت نیازی به درخواست اضافی ندارد
- **تأثیر احتمالی**: پیش‌واکشی سمت سرور زمان رندر SSR را افزایش می‌دهد. هرچه دیکشنری‌های بیشتری پیش‌واکشی شوند، پاسخ سرور کندتر می‌شود
- **توصیه**: فقط نوع‌های دیکشنری ضروری برای اولین نمایش را در `prefetch` پیکربندی کنید (معمولاً ۳-۱۰ عدد)، همه دیکشنری‌ها را قرار ندهید

## نکات

> پیش‌واکشی فقط در حالت SSR کار می‌کند. اگر پروژه شما حالت رندر خالص کلاینت (SPA) است، نیازی به پیکربندی `prefetch` نیست.

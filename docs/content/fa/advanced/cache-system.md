**هدف**: درک معماری کش سه سطحی ماژول دیکشنری، جریان خواندن/نوشتن و تنظیم پیکربندی.

## معماری کش سه سطحی

```
جریان درخواست:
  کامپوننت کاربر useDict / useDict / useDictTree را فراخوانی می‌کند
    ↓
  سطح ۱: کش حافظه (Map) ← سریع‌ترین، در صورت یافتن مستقیماً برمی‌گرداند
    ↓ یافت نشد
  سطح ۲: کش ماندگار IndexedDB ← قابل نگهداری بین صفحات و بستن مرورگر
    ↓ یافت نشد / عدم تطابق نسخه
  سطح ۳: درخواست شبکه API ← منبع نهایی داده
    ↓ دریافت موفق
  بازنویسی: همزمان در IndexedDB + کش حافظه نوشته می‌شود
```

## جزئیات هر سطح کش

### سطح ۱: کش حافظه (Map)

- **محل ذخیره‌سازی**: شیء `Map` در حافظه JavaScript
- **چرخه حیات**: با بستن صفحه پاک می‌شود
- **محدودیت ظرفیت**: حداکثر `cache.memoryMax` ورودی (پیش‌فرض ۲۰۰)
- **استراتژی انقضا**: پس از `cache.ttl` میلی‌ثانیه منقضی می‌شود (پیش‌فرض ۰ = هرگز منقضی نشود)
- **استراتژی حذف**: LRU (کمترین استفاده اخیر)، در صورت عبور از ظرفیت، قدیمی‌ترین ورودی حذف می‌شود
- **سرعت**: بسیار سریع (سطح میکروثانیه)

### سطح ۲: کش ماندگار IndexedDB

- **محل ذخیره‌سازی**: پایگاه داده IndexedDB مرورگر
- **چرخه حیات**: ذخیره‌سازی ماندگار، مگر اینکه کاربر داده‌های مرورگر را پاک کند یا ماژول کش را باطل کند
- **قابل تنظیم**: `cache.indexedDB.enabled` را `false` کنید تا این سطح رد شود
- **سرعت**: سریع (سطح میلی‌ثانیه)، بسیار سریع‌تر از درخواست شبکه

### سطح ۳: درخواست شبکه API

- **منبع داده**: API دیکشنری پیکربندی شده (رابط REST پیش‌فرض یا آداپتور سفارشی)
- **حذف درخواست‌های تکراری**: درخواست‌های همزمان برای یک نوع دیکشنری در یک تماس شبکه ادغام می‌شوند
- **مدیریت شکست**: شکست درخواست یک استثنا پرتاب می‌کند که توسط وضعیت error در composable گرفته می‌شود

## جریان خواندن و نوشتن

### جریان خواندن (getDict)

```
۱. بررسی کش حافظه
   ├─ یافت شد ← بلافاصله برمی‌گرداند (شامل بررسی TTL، منقضی شده ناموجود محسوب می‌شود)
   └─ یافت نشد ← به مرحله ۲

۲. بررسی حذف درخواست تکراری
   ├─ درخواست در حال انجام با همان key وجود دارد ← همان Promise را برمی‌گرداند
   └─ درخواست در حال انجام وجود ندارد ← به مرحله ۳

۳. بررسی IndexedDB
   ├─ یافت شد ← کش حافظه را بازنویسی کرده، داده را برمی‌گرداند
   └─ یافت نشد / IndexedDB فعال نیست ← به مرحله ۴

۴. درخواست شبکه API
   ├─ موفق ← همزمان در IndexedDB + کش حافظه نوشته، داده را برمی‌گرداند
   └─ شکست ← استثنا پرتاب می‌کند
```

### بازنشانی اجباری (refresh)

```
۱. ورودی متناظر در کش حافظه پاک می‌شود
۲. رکورد درخواست در حال انجام پاک می‌شود
۳. مستقیماً درخواست شبکه API ارسال می‌شود (رد کردن IndexedDB)
۴. موفق ← در IndexedDB + کش حافظه نوشته می‌شود
```

## تشخیص نسخه و ابطال خودکار

در اولین دسترسی به داده‌های دیکشنری یک مخزن (هنگام فراخوانی `useDict` / `useDictTree` / `useDict`)، ماژول به صورت تنبل `fetchVersion()` را برای دریافت شماره نسخه راه دور فراخوانی کرده و با شماره نسخه ذخیره شده در localStorage مقایسه می‌کند:

- **تطابق نسخه** ← داده‌های کش معتبرند، مستقیماً استفاده می‌شوند
- **عدم تطابق نسخه** ← تمام کش آن مخزن پاک می‌شود (حافظه + IndexedDB)، درخواست‌های بعدی همه مجدداً دریافت می‌شوند
- **شکست تشخیص نسخه** ← خطا نادیده گرفته می‌شود، کش ادامه می‌یابد

## پیکربندی کش

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@lacqjs/nuxt-dict'],
  dict: {
    cache: {
      memoryMax: 200,       // حافظه حداکثر ۲۰۰ ورودی دیکشنری کش می‌کند
      ttl: 0,               // ۰ = هرگز منقضی نشود، واحد میلی‌ثانیه
      indexedDB: {
        enabled: true,      // آیا IndexedDB فعال شود
        dbName: 'nuxt-dict', // نام پایگاه داده
      },
    },
    version: {
      storageKey: '__NUXT_DICT_VERSION__', // نام کلید localStorage
    },
  },
})
```

### پیشنهادات تنظیم

| سناریو | پیکربندی پیشنهادی |
|--------|-----------|
| داده‌های دیکشنری به ندرت تغییر می‌کنند | `ttl: 0`، متکی بر ابطال شماره نسخه |
| داده‌های دیکشنری مکرراً تغییر می‌کنند | `ttl: 600000` (۱۰ دقیقه انقضا) |
| موبایل، فضای ذخیره‌سازی محدود | `memoryMax: 50`، `indexedDB.enabled: true` |
| بدون نیاز به کش ماندگار | `indexedDB.enabled: false` |

# i18n Internationalization

**Goal**: Use nuxt-dict together with @nuxtjs/i18n so that dictionary data automatically syncs when users switch languages.

## When do you need this?

- Your project already uses @nuxtjs/i18n for multilingual routing and UI translations
- You want dropdown options (gender, status, etc.) to automatically switch to the correct language when users change language
- Dictionary data should update without a page refresh

## Passive Sync (works out of the box)

nuxt-dict's default cookie name is `i18n_redirected` — exactly the same as @nuxtjs/i18n's default cookie. As long as you keep `locale.source: 'cookie'` (the default), both modules automatically share the same language information.

**What does this mean?**

When a user first visits `/en/about`:
1. @nuxtjs/i18n parses `en-US` from the URL and writes it to the `i18n_redirected` cookie
2. nuxt-dict reads the language from the same cookie and requests dictionary data with `lang=en-US`
3. The dictionary API returns English data

When the user refreshes or navigates to a new page:
1. The cookie already contains `en-US`
2. nuxt-dict uses this language directly — no extra configuration needed

## Active Sync (runtime language switch without page refresh)

**This is built into nuxt-dict — no extra code required.**

When @nuxtjs/i18n switches the language (e.g., user clicks a language toggle):

```
@nuxtjs/i18n switches language
  → writes cookie i18n_redirected = 'en-US'
    → nuxt-dict cookie watch detects the change
      → DictManager.setLocale('en-US') → clears cache
        → useDict / useDictTree watch fires
          → automatically re-fetches dictionary data with the new language
```

**Users don't need to write any `watch` or `refresh()` code** — all `useDict`, `useDictTree`, and `useDict` components reload automatically.

## Configuration Example

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@lacqjs/nuxt-dict', '@nuxtjs/i18n'],

  // nuxt-dict config
  dict: {
    api: { baseURL: '', dictEndpoint: '/api/dict/list' },
    locale: {
      source: 'cookie',             // default, no need to change
      cookieKey: 'i18n_redirected', // default, shares the same cookie with @nuxtjs/i18n
      default: 'zh-CN',
    },
  },

  // @nuxtjs/i18n config
  i18n: {
    locales: ['zh-CN', 'en-US', 'ja-JP'],
    defaultLocale: 'zh-CN',
    strategy: 'prefix_except_default',
  },
})
```

## Behavior Under Different Strategies

| @nuxtjs/i18n strategy | URL changes on switch | useDict re-fetches |
|-----------------------|:--:|:--:|
| `prefix_except_default` (default) | Yes (`/about` → `/en/about`) | ✅ Auto |
| `prefix` | Yes (`/zh/about` → `/en/about`) | ✅ Auto |
| `no_prefix` | No | ✅ Auto (via cookie watch) |

Regardless of strategy, dictionary data always syncs — under `prefix_*` strategies, route changes trigger component recreation (natural re-fetch); under `no_prefix`, the cookie watch provides a safety net.

## Notes

> Make sure nuxt-dict's `locale.cookieKey` matches @nuxtjs/i18n's `detection.cookieKey`. If you change one, update the other.

> Tree dictionaries (`useDictTree`) also re-fetch automatically on language switch, but the **currently selected node is not automatically re-selected**. If your cascader stores values as codes, translated labels update automatically. If values are label text, you may need extra handling after a language switch.

> During SSR prefetch, the server fetches data using the initial request's language. If the client language differs after hydration, data will re-fetch automatically. A brief flash of the wrong language on first paint is part of the normal hydration process.
